# 5. MVP 범위

---

## ④ MVP Scope (최종본 v3)

> 변경사항: Core Haptics(Foreground 전용) 관련 항목 전면 삭제. 진동 메커니즘을 "커스텀 사운드 파일 기반 백그라운드 진동" 단일 방식으로 통일.
> 

---

### 1. 필수 기능 (Must Have)

| 기능 | 설명 | 근거 |
| --- | --- | --- |
| 리그 선택 (EPL / World Cup) | 앱 최초 실행 시 지원 리그 중 선택 | 스코프를 좁혀야 데이터/QA 부담이 통제됨 |
| 관심 팀 선택 | 선택한 리그 내 팀 등록 | 개인화의 최소 단위, Push 필터링 기준 |
| 오늘/예정 경기 목록 확인 | 팔로우 가능한 경기 리스트 | 사용자가 "무엇을 등록할지" 알아야 함 |
| 경기 팔로우(등록)/해제 | 특정 경기를 진동 알림 대상으로 지정 | 핵심 트리거 |
| 경기 시작 알림 (진동) | 경기 시작 전/시점 알림 | "따라가고 있음"을 인지하는 첫 순간 |
| 골 진동 알림 | 가장 중요한 이벤트 | 앱의 존재 이유 |
| VAR 진동 알림 | 판독 중 상황 전달 | 축구 특유의 긴장 이벤트, 차별화 포인트 |
| 옐로카드 진동 알림 | 경고 상황 전달 | 흐름 이해에 필요한 최소 이벤트 |
| 레드카드 진동 알림 | 퇴장 상황 전달 | 경기 판도를 바꾸는 핵심 이벤트 |
| 하프타임 진동 알림 | 전반 종료 | "마디" 인지 포인트 |
| 경기 종료 진동 알림 | 풀타임 | 세션의 자연스러운 종료점 |
| **이벤트별 커스텀 사운드 파일 제작 (.caf, 총 7종)** | 골/VAR/옐로/레드/시작/하프타임/종료 각각 다른 파형 설계 | **진동 구분의 유일한 메커니즘** — Core Haptics 대체, 사운드 디자인이 핵심 작업으로 격상 |
| **푸시 payload → 사운드 파일 동적 매핑** | 서버에서 이벤트 타입에 맞는 sound 파일명을 push에 지정 | 백그라운드/종료 상태에서도 시스템이 알아서 파형 기반 진동 재생 |
| Apple Watch 진동 (iPhone 알림 미러링) | 워치 전용 코드 없이, iPhone 알림의 사운드-진동 동기화를 그대로 상속 | 워치 별도 익스텐션 불필요 (아래 5번 참고) |
| 경기 상세 화면 (스코어, 상태, 이벤트 요약 리스트) | 확인하고 싶을 때만 보는 화면 | Screenless 철학 유지, 확인용 화면일 뿐 |
| 햅틱 패턴 가이드 화면 (사운드 기반 진동 미리듣기) | 온보딩/설정에서 언제든 재생 가능 | 학습 보조 |
| 관심 팀/리그 관리 (설정) | 개인화 관리 |  |
| 알림 On/Off (전체 토글) |  |  |
| 축구 API 연동 및 실시간 이벤트 폴링/수신 | 백엔드/인프라 |  |
| 이벤트 → 푸시 발송 파이프라인 (사운드 매핑 포함) | 백엔드/인프라 |  |

### 2. 있으면 좋은 기능 (Nice to Have)

| 기능 | 왜 지금은 아닌가 |
| --- | --- |
| 진동 패턴 커스터마이징 | 먼저 표준 사운드-진동 패턴의 학습 성공률부터 검증 |
| 위험한 공격 알림 | 데이터 신뢰도/판단 기준 모호 |
| 스코어 텍스트 요약 화면 고도화 | 최소 요약이면 충분, 정교한 UI는 후순위 |
| 다크모드 / 위젯 | MVP 검증과 무관 |
| 여러 경기 동시 팔로우 | 패턴 혼동 리스크 |

### 3. 절대 넣지 않을 기능 (Won't Have) — MVP 단계

| 기능 | 배제 이유 |
| --- | --- |
| **Core Haptics 기반 Foreground 전용 커스텀 진동** | **오늘 논의로 최종 배제** — 사용자가 화면을 보며 앱을 켜놓고 기다리는 상황 자체가 제품 철학과 모순. 앱 다운로드 목적이 "화면을 안 보는 것"이므로 이 기능에 리소스를 쓰지 않음 |
| Apple Watch 화면 UI (스코어/타임라인) | Screenless 철학과 정면 충돌 |
| Apple Watch 전용 네이티브 익스텐션 코드 | **오늘 논의로 불필요해짐** — 사운드-진동 동기화가 iPhone 알림 미러링만으로 워치에 자동 전달되므로 별도 워치 코드 자체가 필요 없음 |
| Android 앱 / Galaxy Watch(Wear OS) 지원 | MVP는 iOS 우선, 백엔드는 확장 고려 설계 |
| 커뮤니티/채팅/실시간 응원 기능 | 리소스 분산 |
| 뉴스/기사 피드 | 정보 과부하 방지 원칙과 충돌 |
| 선수별 상세 통계, 히트맵 등 | 컨셉과 무관 |
| 베팅/예측 관련 기능 | 법적 리스크 |
| 다국어 지원 | MVP는 단일 언어 우선 |
| 로그인/회원가입 | 진입장벽 |
| 유료 구독/결제 | 가치 검증이 먼저 |
| 다수 리그 전체 지원 | QA 리소스 집중 |

### 4. 확장성 원칙 (설계 시 반영)

| 원칙 | 이유 |
| --- | --- |
| 푸시 발송은 크로스플랫폼 서비스(FCM 등) 기반 | Android 확장 시 로직 재사용 |
| "이벤트 타입"을 표준화된 값으로 관리 | Android(Wear OS)는 파형 자유도가 더 높으므로, 매핑 로직만 재사용하고 사운드/진동 자체는 추후 더 정교하게 재설계 가능 |

---

### 5. MVP 플랫폼 범위 최종 확정 (v3)

| 구분 | MVP 범위 |
| --- | --- |
| 메인 앱 | React Native (iOS 전용 출시) |
| 진동 메커니즘 | **iPhone/Apple Watch 공통 — 커스텀 사운드 파일의 파형 기반 진동 하나로 통일** |
| 별도 네이티브 개발 | **불필요** (Core Haptics 브릿지 삭제, WatchKit 익스텐션 삭제) |
| 제외 | Android 앱, Galaxy Watch, Tizen 워치 |

> **가장 중요한 변화**: 지난 버전 대비 네이티브 개발 리소스가 크게 줄었습니다. 이제 핵심 리소스는 "Swift 네이티브 코딩"이 아니라 "사운드 파형 디자인 + 서버 매핑 로직"입니다. React Native 개발자 중심으로 MVP를 완주할 수 있는 구조가 되었습니다.
>
