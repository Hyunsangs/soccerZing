# 9. api 아키텍쳐

## ⑨ API Architecture

API-Football을 기준으로 작성합니다. (Sportmonks는 후보로 남겨두되, 아래 구조는 두 서비스 모두에 유사하게 적용 가능합니다.)

---

### 1. 필요한 API 엔드포인트

MVP 기능(④ Scope, ⑦ Feature List)을 기준으로 실제 필요한 API만 추립니다.

| 용도 | API-Football 엔드포인트 (참고) | 호출 시점 |
| --- | --- | --- |
| 리그 정보 조회 | `/leagues` | 최초 세팅 시 1회 (EPL, World Cup ID 확보) |
| 팀 목록 조회 | `/teams` | 리그 선택 시 (팀 선택 화면 구성) |
| 오늘/예정 경기 목록 | `/fixtures?date=` 또는 `/fixtures?league=&season=` | 홈 화면 진입 시, 하루 1~수 회 캐싱 |
| **실시간 경기 이벤트** | `/fixtures/events?fixture=` | **경기 진행 중 반복 폴링 (핵심)** |
| 실시간 스코어/상태 | `/fixtures?live=all` 또는 `/fixtures?id=` | 경기 상세 화면 진입 시, 폴링 병행 |

> MVP에서는 이 5개 엔드포인트 외에는 사용하지 않습니다. 선수 통계, 팀 순위, 부상 정보 등은 명시적으로 제외 (④ Scope의 "절대 넣지 않을 기능"과 일관)
>

### 2. Polling vs WebSocket — 결론: Polling

#### 왜 WebSocket이 아닌가

| 검토 항목 | 내용 |
| --- | --- |
| API-Football 자체 지원 | API-Football은 기본적으로 **REST 기반 Polling 방식**이며, 실시간 WebSocket 스트림을 표준 제공하지 않습니다 (상위 플랜에 Webhook 옵션이 있으나 MVP 예산 고려 시 과도) |
| MVP 원칙 | 가장 단순한 구조로 검증하는 것이 목표. WebSocket 인프라(연결 유지, 재연결 처리, 서버 부하 관리)는 초기 리소스 대비 복잡도가 큼 |
| 이벤트 특성 | 축구 경기 이벤트는 "완전 실시간(밀리초 단위)"까지 필요하지 않음. 골이 몇 초 늦게 감지돼도 사용자 경험에 치명적이지 않음 (문자중계 앱들도 통상 수십 초 지연 존재) |

**결론**: MVP는 **서버가 API-Football을 주기적으로 Polling → 이벤트 감지 시 → 자체 푸시 발송** 구조로 설계합니다.

### 3. 요청 주기 설계

경기 상태에 따라 폴링 주기를 다르게 가져가는 것이 핵심입니다. (상시 고정 주기로 폴링하면 API 비용이 불필요하게 커짐)

| 경기 상태 | 폴링 주기 | 이유 |
| --- | --- | --- |
| 경기 없음 (평상시) | 폴링 안 함, 또는 6시간 간격으로 "오늘 경기 있는지"만 확인 | 불필요한 API 소비 방지 |
| 경기 시작 10분 전 | 1회성 체크, 시작 알림 발송 준비 | 시작 알림 정확도 |
| **경기 진행 중 (Live)** | **15~30초 간격** | 골/카드 등 이벤트를 준실시간으로 감지해야 함, MVP 예산 내에서 현실적인 최소 주기 |
| 하프타임 | 폴링 주기 완화 (60초) | 이벤트 발생 가능성 낮음 |
| 경기 종료 감지 후 | 해당 경기 폴링 즉시 중단 | 리소스 낭비 방지 |

#### 폴링 대상 필터링

- **팔로우된 경기만 폴링** (전체 EPL/WC 경기를 다 폴링하지 않음). 사용자가 아무도 팔로우하지 않은 경기는 API 호출 대상에서 제외
- 단, "오늘 경기 목록"을 홈 화면에 보여줘야 하므로, 기본 스케줄 정보(경기 목록 자체)는 낮은 빈도(예: 3시간 간격)로 별도 캐싱

### 4. Rate Limit 대응 전략

API-Football의 플랜별 호출 제한(예: 무료/저가 플랜은 분당 요청 수 제한이 있음)을 고려한 설계입니다.

#### 4-1. 서버 중앙 집중식 Polling (사용자별 개별 호출 절대 금지)

```
[잘못된 구조 — 하면 안 됨]
사용자 A 앱 → API-Football 직접 호출
사용자 B 앱 → API-Football 직접 호출
사용자 C 앱 → API-Football 직접 호출
→ 사용자 수만큼 API 호출 폭증, Rate Limit 즉시 초과

[올바른 구조]
서버 (1곳) → API-Football 폴링 (경기당 1회)
  ↓
서버가 이벤트 감지
  ↓
서버 → 해당 경기를 팔로우하는 모든 사용자에게 푸시 발송 (Fan-out)
```

이 구조가 Rate Limit 대응의 핵심입니다. **API 호출 횟수는 "동시 진행 경기 수"에만 비례하고, 사용자 수와는 무관**하게 설계됩니다. MVP는 EPL+World Cup만 지원하므로 동시 진행 경기 수 자체가 매우 제한적입니다 (같은 시간대 최대 몇 경기 수준).

#### 4-2. 캐싱

| 데이터 | 캐싱 전략 |
| --- | --- |
| 리그/팀 정보 | 거의 안 바뀜 → 1일~1주 캐싱 |
| 오늘 경기 목록 | 하루 단위 캐싱, 몇 시간마다 갱신 |
| 라이브 이벤트 | 캐싱 안 함 (실시간성 필요), 대신 폴링 주기로 빈도 조절 |

#### 4-3. 요청 실패/초과 대응

- Rate Limit 초과(429 응답) 시 **Exponential Backoff**(재시도 간격을 점진적으로 늘림) 적용
- 폴링 실패가 반복되는 경기는 알림 발송을 건너뛰지 않고, 다음 성공 폴링 시 **누락된 이벤트를 한 번에 비교/감지**하는 로직 필요 (마지막 확인 시점 대비 diff 방식)
- 월간 호출 한도에 근접 시 관리자에게 알림 (운영 리스크 관리, ⑭ Risks에서 재논의)

### 5. 이벤트 감지 로직 (Diffing)

API-Football의 `/fixtures/events` 응답은 "현재까지 발생한 전체 이벤트 리스트"를 반환하는 방식이므로, 서버는 다음과 같이 처리합니다.

```
매 폴링마다:
  1. /fixtures/events 호출 → 전체 이벤트 리스트 수신
  2. 이전 폴링 시점에 저장해둔 이벤트 리스트와 비교 (diff)
  3. 새로 추가된 이벤트만 추출
  4. 새 이벤트 → 이벤트 타입 매핑 (골/VAR/카드 등)
  5. 매핑된 타입에 맞는 사운드 파일명 결정 (⑧ Haptic Design 참고)
  6. 해당 경기 팔로워 목록 조회 → 푸시 발송 (Fan-out)
  7. 현재 이벤트 리스트를 다음 비교를 위해 저장
```

> VAR의 경우 API-Football 데이터상 "판독 시작"과 "판독 결과 확정"이 별도 이벤트로 오는지, 단일 이벤트로 오는지 실제 API 스펙 확인이 필요합니다. 이 부분은 개발 착수 전 **API 응답 샘플을 먼저 확인하는 스파이크 작업**으로 처리하는 것을 권장합니다 (⑧ Haptic Design의 실기기 검증과 함께 1주차에 병행).
>

### 6. 전체 아키텍처 흐름도

```
┌─────────────────┐
│   API-Football    │
└─────────┬─────────┘
          │ Polling (15~30초 간격, 팔로우된 경기만)
          ▼
┌─────────────────────────┐
│   서버 (Polling Worker)    │
│   - 이벤트 diff 감지          │
│   - 이벤트 타입 → 사운드 매핑    │
└─────────┬─────────────────┘
          │ Fan-out
          ▼
┌─────────────────────────┐
│   푸시 발송 서비스 (FCM 등)     │
└─────────┬─────────────────┘
          │
          ▼
┌─────────────┐      ┌──────────────────┐
│  iPhone 알림   │─────▶│  Apple Watch (미러링) │
│  (사운드 파형 재생) │      │  (진동 자동 전달)      │
└─────────────┘      └──────────────────┘
```

### 7. API-Football vs Sportmonks 최종 판단 (참고)

| 기준 | API-Football | Sportmonks |
| --- | --- | --- |
| 이벤트 데이터 상세도 | 골/카드/VAR 이벤트 지원 확인됨 | 유사 수준, 플랜별 상이 |
| 가격 구조 | 무료 플랜 존재, 유료 플랜 합리적 | 상대적으로 진입 장벽 있음 |
| MVP 적합성 | 무료/저가 플랜으로 검증 가능 | 초기 검증 단계엔 다소 무거움 |

**MVP는 API-Football 우선 추천**, 단 실제 계약 전 **VAR 이벤트 데이터의 정확도/지연시간을 무료 플랜으로 직접 테스트**해보는 것을 권장합니다 (VAR은 우리 제품의 차별화 포인트인데 데이터 소스 신뢰도가 검증 안 됐다면 리스크).
