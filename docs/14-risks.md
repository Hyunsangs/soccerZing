# 14. 리스크

## ⑭ Risks

### 1. 기술적 리스크

| 리스크 | 설명 | 영향도 | 대응 |
| --- | --- | --- | --- |
| **사운드 파형 기반 진동의 신뢰성** | 백그라운드/종료 상태에서 이벤트별 진동 구분이 기대만큼 명확하지 않을 수 있음 (⑧ 문서에서 확인된 최대 리스크) | 🔴 매우 높음 (제품 핵심 가정) | 1주차 최우선 기술 스파이크로 조기 검증, 실패 시 파형 재설계 |
| **워치 사운드 실제 재생 문제** | 무음 모드 미설정 시 소리가 실제로 들려 "몰래 확인" 시나리오가 깨질 수 있음 | 🟠 높음 | 온보딩 무음모드 안내(⑤ 문서 신규 스텝), 사운드 파일 자체 저음량 설계 |
| **VAR 이벤트 데이터 정확도/지연** | API-Football이 VAR 판독 시작/종료를 얼마나 정확하고 빠르게 제공하는지 미검증 | 🟠 높음 | 1주차 API 스파이크로 실측, 부정확 시 VAR을 P1로 하향 조정 검토 |
| **iOS Focus 모드로 인한 알림 지연/누락** | 사용자가 방해금지/회의 모드 등을 켜두면 알림 자체가 늦게 오거나 안 올 수 있음 | 🟡 중간 | 온보딩에서 Focus 모드 예외 앱 등록 안내 추가 검토 (P1) |
| Rate Limit 초과 | 동시 진행 경기가 예상보다 많거나 폴링 주기 설정 오류 시 API 호출 초과 | 🟡 중간 | 서버 중앙 폴링 구조로 근본 대응 (⑨ 문서), Exponential Backoff |

### 2. UX 리스크

| 리스크 | 설명 | 대응 |
| --- | --- | --- |
| 패턴 학습 실패 | 사용자가 패턴을 못 외워 결국 화면을 계속 켜서 확인하게 됨 (③ 문서 C 흐름의 "실패 상태") | 햅틱 가이드 화면으로 반복 학습 유도, 블라인드 테스트로 사전 검증 |
| 알림 피로도 | 90분 내내 여러 이벤트 알림이 오면 오히려 피곤함을 느낄 수 있음 | ⑧ 문서 "피로도 관리" 원칙에 따라 이벤트별 강도 차등, 사소한 이벤트는 절제 |
| 온보딩 이탈 | 무음모드 안내 등 스텝이 늘어나며 온보딩 완주율이 떨어질 위험 | 조건부 노출(워치 미착용 시 스킵)로 최소화, 전체 스텝 수 지속 모니터링 |

### 3. API 비용 리스크

| 리스크 | 설명 | 대응 |
| --- | --- | --- |
| 무료/저가 플랜의 한도 초과 | 베타 사용자가 예상보다 많거나, 동시 경기 수가 예상보다 많을 경우 | 팔로워 있는 경기만 폴링(⑨ 문서)으로 근본적 절감, 월 한도 근접 시 수동 모니터링 |
| 플랜 업그레이드 필요성 | VAR 등 상세 이벤트 데이터가 상위 플랜에서만 제공될 가능성 | 1주차 API 스파이크에서 필요한 플랜 등급 사전 확인 |

### 4. 배터리 리스크

| 리스크 | 설명 | 대응 |
| --- | --- | --- |
| 백그라운드 푸시 자체는 배터리 영향 적음 | 우리 앱은 클라이언트가 폴링하지 않고 서버가 폴링 후 푸시만 보내는 구조라, 클라이언트 측 배터리 부담은 일반 푸시 알림 앱 수준 | 별도 대응 불필요 (설계 자체로 이미 리스크 낮음) |
| 사운드 파일 재생 빈도 | 경기 중 잦은 알림이 배터리/성능에 미미하게 영향 가능 | MVP 단계에서는 심각하지 않음, 정식 출시 전 실측 권장 |

### 5. Push Notification 리스크

| 리스크 | 설명 | 대응 |
| --- | --- | --- |
| APNs 인증서/키 설정 오류 | 푸시 발송 실패의 가장 흔한 원인 | 1주차 Firebase 세팅 단계에서 조기 검증 |
| 사용자의 알림 권한 거부 | 거부 시 앱의 핵심 가치 전달 자체가 불가능 | 온보딩에서 권한의 중요성을 명확히 설명(④ 문서), 거부 시 재유도 안내(P1) |
| 커스텀 사운드 파일의 iOS 정책 준수 | 알림 사운드 길이 등에 Apple 가이드라인 제약 존재 가능 | 1주차 스파이크에서 실기기 테스트로 함께 확인 |

---

### 리스크 종합 코멘트

가장 눈여겨봐야 할 것은 **1번 표의 상위 3개 리스크가 전부 "제품이 성립하느냐"에 직결된다는 점**입니다. 특히 "사운드 파형 기반 진동의 신뢰성"은 이 프로젝트의 전제 자체이므로, 릴리즈 플랜에서 이미 반영했듯 **다른 모든 작업보다 우선해서 검증**해야 합니다. 만약 1주차 스파이크에서 이 리스크가 현실화된다면(구분이 잘 안 된다면), MVP 방향 자체를 재검토해야 할 수도 있다는 점을 미리 인지하고 있는 게 좋습니다.
