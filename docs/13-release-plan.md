# 13. 릴리즈 플랜

## ⑬ Release Plan

3주 단위로 계획하되, **1주차 최우선 순위는 화면 개발이 아니라 기술 스파이크(검증)**입니다. 이 순서를 지키지 않으면 2~3주차 작업이 검증 안 된 가정 위에 지어질 위험이 있습니다.

---

### 1주차: 기술 스파이크 + 기초 설계 확정

| 일자 | 작업 | 비고 |
| --- | --- | --- |
| Day 1~2 | **햅틱 기술 스파이크** — 사운드 파일 3~4종 시제품 제작, 실기기(iPhone+Apple Watch, 앱 완전 종료 상태)에서 블라인드 테스트 | ⑧ Haptic Design 문서 7번 검증 계획 실행, **가장 먼저 해야 함** |
| Day 2~3 | **API 스파이크** — API-Football 실제 응답 샘플 확인 (특히 VAR 이벤트 구조), Rate Limit 실측 | ⑨ 문서 5번 참고 |
| Day 3~4 | Firebase 프로젝트 세팅 (Firestore, FCM, Anonymous Auth 초기 구성) | ⑩ 문서 기반 |
| Day 4~5 | RN 프로젝트 초기 세팅, 폴더 구조 적용, 네비게이션 뼈대 구성 | ⑫ 문서 기반 |

> **Go/No-Go 체크포인트**: 1주차 종료 시점에 "화면을 안 보고도 골/카드/VAR을 구분할 수 있는가"가 검증되어야 합니다. 실패 시 ⑧ 문서 재설계 후 1주차를 연장해야 하며, 2주차 화면 개발을 무리하게 시작하지 않습니다.
>

### 릴리즈 플랜 설계 원칙

- **1주차는 "만들기"가 아니라 "검증하기"에 집중**합니다. 화면 UI보다 햅틱 검증과 API 검증이 먼저입니다.
- 2~3주 안에 정식 스토어 출시까지 목표하지 않고, **TestFlight 클로즈드 베타**를 3주차 종료 시점 목표로 잡았습니다. ①Vision에서 정의한 "한 명이라도 '이거 진짜 편하다'를 느끼는 순간"을 베타 사용자에게서 확인하는 것이 이 릴리즈의 실제 목표입니다.
