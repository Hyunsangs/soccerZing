# 8. 햅틱디자인 ( 핵심 )

## ⑧ Haptic Design (최종본 v2)

> 이 프로젝트에서 가장 중요한 문서입니다. 오늘 논의로 메커니즘이 확정되었습니다: **모든 진동은 "이벤트별 커스텀 사운드 파일(.caf)의 파형"을 통해 구현됩니다.** Foreground 전용 Core Haptics는 제품 철학과 맞지 않아 완전히 배제합니다.
> 

---

### 1. 설계 철학 (Design Principles)

| 원칙 | 설명 |
| --- | --- |
| **구분성 (Distinctiveness)** | 인접 이벤트(옐로 vs 레드)는 리듬 구조 자체가 달라야 함. 강도보다 "패턴의 모양"으로 구분 |
| **암기 가능성 (Memorability)** | 하나의 문법(grammar) 안에서 모든 패턴이 파생되어야 함 |
| **중요도 매핑 (Weight Mapping)** | 골 > 레드카드 > VAR/옐로카드 > 시작/하프타임/종료 순으로 강도·길이 배정 |
| **오인 최소화 (Error Minimization)** | 가장 헷갈리기 쉬운 쌍(옐로 vs 레드, 시작 vs 종료)은 리듬을 반대 구조로 설계 |
| **파형 우선 설계 (Waveform-First)** | 코드로 강도를 세밀 제어하는 게 아니라, **사운드 파일 자체의 파형(길이/간격/피크)**으로 모든 구분을 만들어야 함 — 이게 오늘부터 이 문서의 새로운 전제 |

---

### 2. 진동 문법 (Haptic Grammar)

사운드 파일 설계 시에도 동일한 3단위 문법을 그대로 사용합니다. 이 문법이 곧 **오디오 편집 시 파형의 설계도**가 됩니다.

| 기호 | 의미 | 오디오 파형 상의 길이 |
| --- | --- | --- |
| `•` (짧은 진동) | Tap | 약 0.15초, 짧고 뾰족한 피크 |
| `━` (긴 진동) | Hold | 약 0.6~0.8초, 지속되는 낮은 웨이브 |
| `(pause)` | 정지 구간 | 무음 약 0.2~0.3초 |

---

### 3. 이벤트별 패턴 설계 및 사운드 파일 스펙

| 이벤트 | 패턴 표기 | 사용자 언어 | 파일명 (예시) | 오디오 파형 설계 스펙 |
| --- | --- | --- | --- | --- |
| **골 (Goal)** | `━━` | "묵직하게 한 번 크게" | `goal.caf` | 단일 지속음, 길이 0.7초, 볼륨/진폭 최대치. 다른 어떤 이벤트와도 겹치지 않는 "유일한 단일 Hold" 파형 |
| **VAR** | `• ━ •` | "판독 들어감... 결과 나옴" | `var.caf` | 짧은 피크(0.15s) → 무음(0.2s) → 중간 지속음(0.5s) → 무음(0.2s) → 짧은 피크(0.15s), 총 3단 구성 |
| **옐로카드 (Yellow)** | `• •` | "톡톡, 경고예요" | `yellow_card.caf` | 짧은 피크 2회, 간격 0.15s, 볼륨 중간 |
| **레드카드 (Red)** | `• • •━` | "톡톡톡, 그리고 무겁게" | `red_card.caf` | 짧은 피크 3회(간격 0.12s) + 마지막에 지속음 0.6초(볼륨 최대) 이어붙임. 옐로카드 파형의 "확장판" 구조 |
| **경기 시작 (Kick-off)** | `• • •` (점점 빨라짐) | "준비, 준비, 시작!" | `kickoff.caf` | 짧은 피크 3회, 간격이 0.3s → 0.2s → 0.1s로 점점 좁아지는 가속 구조 |
| **하프타임 (Half-time)** | `━ •` | "쉬어가는 지점" | `halftime.caf` | 중간 지속음 0.5초 + 짧은 피크 1회(여운) |
| **경기 종료 (Full-time)** | `• • • ━━` (점점 느려지다 길게) | "천천히 마무리" | `fulltime.caf` | 짧은 피크 3회(간격 점점 증가) + 마지막 지속음 0.6초(볼륨 최대) |

---

### 4. 혼동 방지 매트릭스

| 쌍 | 파형 구조 비교 | 혼동 위험 | 대응 |
| --- | --- | --- | --- |
| 골 vs 하프타임 | 단일 Hold(0.7s) vs Hold+짧은여운 | 낮음 | 골은 끊김 없는 단일 파형, 하프타임은 뒤에 짧은 tap이 붙어 구조가 다름 |
| 옐로 vs 레드 | 피크 2회 vs 피크 3회+Hold | 낮음 | 반복 횟수와 마지막 Hold 유무로 이중 구분 |
| 시작 vs 종료 | 3회 가속 vs 3회 감속+Hold | 낮음 | 종료가 명백히 더 길고 마지막 Hold로 "끝"이라는 감각 부여 |
| VAR vs 골 | 3단 구성 vs 단일 Hold | 매우 낮음 | 구조 자체가 완전히 다름 |

### 5. 구현 메커니즘 (플랫폼 공통, 단일 방식)

지난 버전에서 존재했던 "iPhone Foreground / Background / Watch" 3갈래 구현 방식을 오늘 논의로 **하나로 통일**합니다.

```
[서버] 이벤트 발생 감지   ↓[서버] 이벤트 타입에 맞는 사운드 파일명 결정   (예: 골 → "goal.caf")   ↓[푸시 발송] APNs payload에 sound: "goal.caf" 포함   ↓[iOS 시스템] 알림 수신, 지정된 사운드 파일 재생   ↓[iOS 시스템] 사운드 파형에 동기화된 진동을 iPhone에서 재생   ↓[Apple Watch] iPhone 알림을 자동 미러링, 동일한 파형 기반 진동 재생   (별도 워치 코드 불필요)
```

#### 사전 준비 (개발 시 필요한 것)

1. `goal.caf`, `var.caf`, `yellow_card.caf`, `red_card.caf`, `kickoff.caf`, `halftime.caf`, `fulltime.caf` — 총 7개 사운드 파일을 오디오 편집 툴로 제작해 앱 번들에 포함
2. 서버(푸시 발송 로직)에서 이벤트 타입 → 파일명 매핑 테이블 관리
3. (선택) 사용자가 무음 모드를 선호할 경우를 대비해, 파일 자체는 "귀로 들었을 때는 최소한의 소리, 손목/주머니에서는 뚜렷한 진동"이 되도록 음량은 낮고 파형의 진폭만 큰 방식으로 튜닝

---

### 6. 사용자가 외우는 방식 (Mental Model)

> **"짧은 건 가벼운 상황, 긴 건 무거운 상황. 반복 횟수가 많을수록 심각한 상황."**
> 
- 짧게 2번 = 경고 (옐로)
- 짧게 3번 + 무겁게 = 심각한 경고 (레드)
- 길게 한 번 = 가장 큰 사건 (골)
- 짧게 여러 번 상승 = 뭔가 시작됨
- 짧게 여러 번 하강 + 무겁게 마무리 = 뭔가 끝남

이 문장을 온보딩 카피와 햅틱 가이드 화면에 그대로 사용합니다.

---

### 7. 검증 계획 (최우선 기술 스파이크)

> 이 문서의 모든 설계는 아래 검증을 통과해야 실제로 유효합니다. **개발 착수 전, 다른 어떤 화면 작업보다 먼저 진행합니다.**
> 
1. 위 7개 사운드 파일 중 최소 3~4개(골/VAR/옐로/레드)를 시제품으로 제작
2. 실제 iPhone + Apple Watch 페어링 환경에서 **앱을 완전히 종료한 상태**로 푸시 발송 테스트
3. 화면을 보지 않고 진동만으로 이벤트를 맞히는 블라인드 테스트 (최소 5~10회 반복)
4. 특히 가장 혼동 위험이 높은 "골 vs 레드카드" 쌍을 우선 검증
5. 정확도가 낮게 나오는 파형은 즉시 재설계 (간격/길이/볼륨 조정)

---

### 8. 향후 확장 메모

- 진동 패턴 커스터마이징(P2) 도입 전 반드시 이 표준 사운드-진동 패턴의 학습 성공률을 먼저 검증
- Android/Wear OS 확장 시, `VibrationEffect.createWaveform()`으로 더 정밀한 파형 제어가 가능하므로 오히려 iOS보다 정교한 버전으로 업그레이드 가능. 단 이때도 본 문서의 "문법 구조"(반복 횟수, 상승/하강)는 그대로 유지해 플랫폼 간 일관성 확보
